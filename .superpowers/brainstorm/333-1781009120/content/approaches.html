实现策略方案对比

三种不同的实现组织方式

A

渐进式（推荐）

策略：按依赖关系和影响范围排序，逐功能实现并测试。

顺序：主题系统 → 点击穿透 → 打字机字幕 → 毛玻璃输入框 → 托盘菜单 → 设置界面

理由：主题系统是视觉基础（CSS 变量重构），后续所有 UI 都依赖它。点击穿透是最小的独立改动。打字机和毛玻璃改动中等。托盘和设置依赖前面的 UI 组件。

优势

  • 每步可验证，风险低
  • 后续功能建立在已有基础上
  • 出现问题时容易定位

劣势

  • 需要多次切换上下文
  • 总时间可能稍长
B

分组式

策略：分成两个大块——UI 基础设施 + 功能增强，每组内并行开发。

分组：

第一组（UI 基础）：主题系统 + 点击穿透 + 打字机字幕 + 毛玻璃 → 一次性改造 pet 窗口

第二组（功能）：设置界面 + 托盘菜单 → 统一的应用配置入口

优势

  • 分两波交付，节奏清晰
  • 组内改动可互相配合

劣势

  • 每组改动量大，容易引入 bug
  • 组内同时改动多文件容易冲突
C

新建重构式

策略：创建一个新的 pet-v2 模式窗口，在新窗口中实现所有改进，保留旧窗口作为 fallback。

理由：完全不影响现有功能，可以大胆重构。开发完成后一键切换到 v2。

优势

  • 零风险，不影响现有功能
  • 可以彻底重构架构
  • A/B 对比测试方便

劣势

  • 大量代码重复
  • 维护两套窗口
  • 过度工程化
