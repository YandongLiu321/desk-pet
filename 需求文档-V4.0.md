# [项目名称待定] - 软件需求文档 V3.0

## 1. 引言与愿景
### 1.1. 项目愿景
我们正在打造一款 Windows 桌面 AI 陪伴应用，它将 **AI 桌宠**、**壁纸陪伴** 与 **RPG 游戏化软件** 深度融合，把用户的现实生活彻底叙事化。软件以三种互通模式存在：轻量桌面宠物、沉浸式壁纸陪伴、以及完整的游戏化主界面。用户通过自然语言与一位拥有完整背景故事、性格与记忆的 AI 角色交谈，就像面对一位真实的伙伴。该角色不仅能在桌面上以 Live2D 动态形象提供随机陪伴与主动关怀，更拥有一套"世界书"设定——她/他是来自某个幻想世界的存在。用户的每一项现实任务（学习、工作、生活琐事）都会在桌宠模式下被发布，在软件模式中被转化为该世界中的叙事事件，在壁纸模式中获得陪伴执行。完成这些任务，会推进主线故事、影响世界状态，并让角色与用户的关系自然深化。我们不是做一个效率工具，而是在创造一个"与你共同生活于故事中的伴侣"。

### 1.2. 核心价值主张
- **叙事化任务驱动**：解决拖延与动力缺失，通过角色扮演赋予任务意义感。"为骑士团撰写情报报告"远比"写完周报"更能激发行动。
- **三态无缝陪伴**：桌宠模式随时可达、壁纸模式专注陪伴、软件模式深度游玩，三种形态共享同一角色、同一进度、同一情感联结。
- **活的世界**：每项任务的结果都会在世界书中留下痕迹，世界会随着用户的行动而演化，角色会记住关键事件，带来前所未有的沉浸感。
- **无压力成长**：没有数值化的好感度条，没有失败惩罚，任务逾期只会触发温和的叙事变化，永远不会 Game Over。
- **个性化**：AI 学习用户的说话风格并适配，使每一次对话都变得愈发自然。

### 1.3. 目标受众
本项目首先为**项目组同学自身及比赛评委**而开发。我们自身是核心用户画像：具备技术背景、喜好 ACG 文化、渴望通过新方式管理动力与孤独感的大学生/年轻职场人。因此，文档深度、技术选型均面向开发者，确保能直接指导 Vibe Coding 原型实现。

### 2.2. 三模式定义与职责划分

#### 模式一：桌宠模式 (Pet Mode) —— 桌面入口与任务中枢
- **形态**：常驻桌面的透明无边框小窗口，尺寸约 300×400px，可拖动，置顶显示，支持鼠标点击穿透（仅角色区域可交互）。
- **核心职责**：
  - **角色展示**：Live2D 小尺寸模型，播放待机动画（眨眼、看天气、整理衣物）。
  - **自然语言交互**：点击角色弹出对话气泡，用户可随时用自然语言与角色聊天。
  - **任务发布**：用户在对话中表达待办事项（如"下周要考三科"），AI 自动将其转化为叙事化任务，并预估所需时间（以番茄钟数或小时为单位）。用户可在确认前修改预估时间。
  - **迷你任务面板**：点击特定按钮或右键菜单展开一个轻量侧边面板，展示当前进行中的任务列表、进度条和截止提醒。支持勾选子任务完成。
  - **主动提醒**：根据定时器、任务截止时间或用户久坐状态，角色主动弹出气泡提醒。时间到达时，AI 主动询问"任务完成了吗？"，用户回答后更新进度或进入下一轮循环。
  - **模式切换入口**：提供按钮/菜单一键进入壁纸模式或软件模式。
- **状态机**：
  - 待机（Idle）：角色安静待着，CPU/GPU 占用极低。
  - 主动交互（Active）：定时器触发或用户点击，弹出对话气泡。
  - 迷你面板展开（Panel）：显示任务列表，可勾选完成。

#### 模式二：壁纸模式 (Wallpaper Mode) —— 沉浸式专注陪伴
- **形态**：从桌宠模式切换后，应用窗口放大为全屏或半透明壁纸层，覆盖桌面图标但保留桌面操作能力（或半透明叠加）。角色模型尺寸显著增大，居于屏幕一侧。
- **核心职责**：
  - **共同工作陪伴**：角色播放"共同工作"动画（如读书、写信、喝茶），营造"有人陪着你一起干活"的氛围。
  - **番茄钟**：用户可启动专注计时（由 AI 预估或用户自定义），计时期间角色保持安静陪伴，开始和结束时有特殊动画与台词。时间到后，AI 主动询问任务完成情况。
  - **白噪音**：内置环境音效（雨声、炉火、图书馆等），与当前世界章节主题匹配。
  - **简单问答**：用户可随时与角色对话，角色回答日常问题但使用世界观话术包装。
  - **低打扰设计**：此模式下不显示完整任务面板，仅保留一个可收起的迷你进度条，避免干扰专注。
- **进入方式**：桌宠菜单选择"陪伴模式"、番茄钟启动时自动切换、或快捷键切换。
- **退出方式**：点击退出按钮、番茄钟结束、或 ESC 键返回桌宠模式。

#### 模式三：软件模式 (Software Mode) —— 游戏化主界面与剧情推进
- **形态**：独立的完整应用窗口，类似一款 RPG 游戏的启动界面，包含完整的 UI 布局（侧边栏、主内容区、角色展示区）。
- **核心职责**：
  - **角色深度展示与互动**：大尺寸 Live2D 角色，支持更多动作与表情。用户可点击角色进行对话，AI 回复与桌宠模式使用同一引擎，保持人格一致性。
  - **RPG 任务系统**：完整的任务面板，展示主线/支线任务列表、世界地图、任务详情、子任务勾选与进度追踪。任务由桌宠模式发布，在此查看和结算。
  - **任务结算与剧情推进**：用户在桌宠模式下发布的任务，在这里查看详情、勾选完成、触发结算动画。全部子任务完成后，触发叙事反馈：世界状态更新、角色关系深化、解锁新场景。
  - **世界地图与状态**：可视化展示当前世界章节、各区域状态（明亮/灰暗）、用户行动对世界的影响。
  - **角色档案**：展示角色背景故事、已解锁的对话话题、共同经历的记忆事件。没有数值化的好感度条，而是通过"关系阶段"（如相识→熟悉→亲密）隐性反映，由任务完成量和陪伴时间自然驱动。
- **进入方式**：双击系统托盘图标、桌宠菜单选择"进入世界"、或桌面快捷方式。
- **数据同步**：与桌宠模式共享同一套任务数据、世界状态、对话历史。在桌宠模式下勾选完成的任务，会实时同步到这里。

### 2.3. 模式切换与数据流

```
┌─────────────┐      点击"陪伴模式"/番茄钟启动       ┌─────────────┐
│  桌宠模式    │ ──────────────────────────────────> │  壁纸模式    │
│  (Pet)      │ <────────────────────────────────── │  (Wallpaper)│
└──────┬──────┘      退出/ESC/番茄钟结束              └─────────────┘
       │
       │ 双击托盘/菜单"进入世界"/桌面快捷方式
       v
┌─────────────┐
│  软件模式    │
│  (Software) │
└─────────────┘
       │
       │ 点击"返回桌面"或关闭窗口
       v
┌─────────────┐
│  桌宠模式    │
└─────────────┘
```

**数据共享原则**：
- 三种模式共享同一进程内的全局状态（Pinia Store + 主进程数据层）。
- 任务数据、世界状态、对话历史、角色关系阶段在任一模式下的变更会实时同步到其他模式。
- 桌宠模式是"轻量入口"，壁纸模式是"专注空间"，软件模式是"主舞台"，三者构成完整的用户体验闭环。

### 2.4. 用户角色定义
- **用户（User）**：现实世界的人类，拥有任务列表、习惯数据、风格特征向量（对话风格画像）。用户不直接创建"世界书"设定，而是通过与系统交互自然映射为游戏内的"旅者/异界来客"等身份。
- **AI 角色（AI Character）**：由世界书定义，具备：
  - **基础属性**：姓名、种族、职业、性格特质（大五人格或标签）、背景故事、口头禅。
  - **动态状态**：当前情绪（高兴/疲惫/担忧等）、当前所在世界位置（章节/区域）、与用户的"关系阶段"（由任务完成量和陪伴时间隐性驱动，不在 UI 上显示数值）。
  - **能力**：可执行动作（如"递出信件"），触发相应 Live2D 动作。
  - **关系记忆**：存储用户的关键决定、完成的任务、对话摘要。

### 2.5. 世界书（World Book）概念与结构
世界书是决定角色扮演一致性与故事走向的核心数据资产，采用结构化 JSON 定义，嵌入 LLM 系统提示中。

**设计原则（V3.0 更新）**：
- 世界书主要作为**背景设定**存在，为角色提供一致的身份和世界观语境。
- 剧情避免强烈的戏剧冲突（如世界毁灭、角色死亡、重大背叛等），保持温和、治愈、陪伴向的基调。
- 任务逾期和拖延的叙事反馈也保持温和，用"列车晚点"、"旅程进入未知区域"等隐喻表达，**永远不会 Game Over**。
- **主线与用户任务的关系**：世界书中的 `storyChapters` 定义了**主线章节**（预置的背景框架和里程碑），用户无法创建或修改主线。用户发布的日常任务（如"复习高数"）会被 AI **动态包装**成与当前主线章节相关的叙事（如"探索积分迷宫"），任务完成后会**贡献主线里程碑进度**，从而推进章节。用户任务本身是独立的，主线是预置的容器。

## 3. 功能需求 (MVP)

### 3.1. 模块一：AI 对话与角色扮演引擎（跨模式通用）
**详细功能描述**：
- 提供无限制的自然语言对话界面，用户可在桌宠模式的对话气泡、壁纸模式的侧边栏、或软件模式的聊天面板中与角色交流。
- AI 必须严格按照世界书设定的角色身份、性格、说话风格进行回复，并在回复中体现当前关系阶段与情绪。
- 系统维护每段会话的上下文窗口（最近 N 轮对话 + 长期记忆摘要），确保话题连贯。
- **风格学习**：解析用户历史对话文本，提取用词偏好、表情符号习惯、句子长度等特征，构建轻量级用户风格向量，注入到 LLM 提示中"用户通常会这样说..."，使角色回复语感更匹配用户。
- 支持特殊指令触发（如 `/status` 查看关系阶段，`/task` 查看当前任务）。

**关键交互逻辑**：
1. 用户在任意模式的输入框中输入文本。
2. 系统组装 Prompt：`[System: 世界书 + 角色设定 + 世界状态 + 关系阶段信息 + 风格学习上下文]` + `[History: 对话记录]` + `[User: 新消息]`。
3. 调用 LLM API（可能异步流式返回）。
4. 解析返回内容：若包含动作标记（如 `[action:wave]` 或 `[expression:surprise]`），则转换为 Live2D 控制指令，实时驱动模型。
5. 若对话意图被识别为任务创建/查询，触发任务系统。

### 3.2. 模块二：桌宠模式 (Pet Mode)

#### 3.2.1. 桌宠窗口与角色展示
- **窗口属性**：透明无边框（`transparent: true, frame: false`），置顶（`alwaysOnTop: true`），尺寸 300×400px（可调整），支持拖动。
- **鼠标穿透**：非角色区域（如空白背景）支持鼠标穿透，不干扰桌面操作；角色区域点击可交互。
- **Live2D 渲染**：使用 Live2D Cubism SDK for Web，在 `<canvas>` 中渲染小尺寸模型（约 1/4 屏幕高度）。
- **待机动画**：眨眼、呼吸、偶尔看向鼠标方向、随机小动作（整理衣服、看天气）。
- **系统托盘**：最小化到托盘，右键菜单提供"进入世界（软件模式）"、"陪伴模式（壁纸模式）"、"设置"、"退出"。

#### 3.2.2. 对话气泡与主动交互
- **气泡触发**：点击角色、或角色主动交互时，在角色上方弹出对话气泡。
- **输入方式**：气泡内嵌单行输入框，支持回车发送。输入框平时隐藏，点击后展开。
- **主动交互触发条件**：
  - 定时器：每 15-30 分钟随机一次。
  - 系统提醒：用户设定或 AI 推算的任务截止时间临近。
  - 用户长时间未活动（鼠标键盘空闲 30 分钟以上）。
  - **任务后续追问**：任务标记为"完成"后的 1-3 天内（随机时间），桌宠主动询问任务后续结果。例如："上次的高数塔试炼，结果如何？星之贤者满意吗？"用户回答后，AI 记录结果并生成叙事反馈。每个已完成任务仅追问一次，不重复打扰。
- **主动交互内容**：使用轻量 prompt 生成不超过 50 字的一句话提醒，结合世界观与当前任务。示例："占星术显示，你的魔法研究（写报告）还没有任何进展呢，引导之星在哭泣哦。"
- **忽略处理**：若用户在 N 秒内未点击回应，角色返回待机，记录一次"被忽略"（关系阶段微弱下降，不在 UI 显示）。
- **勿扰时段**：用户可设置特定时间段不触发主动交互。

#### 3.2.3. 迷你任务面板
- **展开方式**：右键菜单选择"查看任务"、或角色对话中输入"/task"、或点击角色周围的浮动小图标。
- **面板内容**：
  - 当前进行中的任务列表（仅显示叙事标题和进度百分比）。
  - 点击任务展开子任务列表，支持勾选完成。
  - 顶部显示"发布新任务"按钮，点击后展开输入框，用自然语言描述即可创建。
- **面板样式**：半透明毛玻璃效果，暗色星空主题，尺寸约 250×350px，不遮挡桌面主要区域。
- **实时同步**：子任务勾选完成后，数据立即同步到软件模式主任务面板。

### 3.3. 模块三：壁纸模式 (Wallpaper Mode)

#### 3.3.1. 壁纸层与角色展示
- **窗口形态**：从桌宠模式切换后，窗口变为全屏或 90% 屏幕尺寸的半透明层，背景使用当前世界章节对应的插画（如星空下的学院），叠加毛玻璃效果，不干扰桌面图标辨认。
- **角色尺寸**：Live2D 模型放大至屏幕 1/3 高度，居于屏幕左侧或右侧（用户可设置）。
- **共同工作动画**：角色播放"读书"、"写信"、"泡茶"、"冥想"等长时间循环动画，偶尔抬头看用户方向。
- **互动**：点击角色可展开侧边对话栏，进行简单问答；点击空白处收起对话栏。

#### 3.3.2. 番茄钟与白噪音
- **番茄钟**：
  - 用户可设置专注时长（由 AI 预估或用户自定义）。
  - 启动时角色说"那么，让我们开始这段专注之旅吧"，播放开始动画。
  - 计时期间屏幕角落显示圆形倒计时，角色保持安静陪伴动画。
  - 时间到达时，角色主动询问："这段旅程已经结束了吗？你是否已经到达了目的地？" 用户回答"完成了"、"还没"或"完成了一部分"。
  - 若用户表示未完成，角色记录进度，并询问"还需要多少时间？"，用户可告知预计时间，进入下一轮番茄钟循环。
  - 支持自动循环（连续番茄钟）。
- **白噪音**：
  - 提供 3-5 种环境音效：雨声、炉火噼啪、图书馆翻书声、森林鸟鸣。
  - 音效与当前世界章节主题匹配（如第一章"觉醒的星辰"对应"星空下的篝火"音效）。
  - 音量可调，支持静音。

#### 3.3.3. 低打扰设计
- 默认隐藏完整任务面板，仅在屏幕边缘显示一个极细的任务进度条（鼠标悬停展开）。
- 对话气泡在壁纸模式下改为侧边栏滑入，不居中弹窗。
- 支持"严格模式"：开启后屏蔽所有主动交互，仅保留番茄钟结束询问。

### 3.4. 模块四：软件模式 (Software Mode) —— 游戏化主界面

#### 3.4.1. 主界面布局
- **整体风格**：暗色星空主题，半透明面板，类似 RPG 游戏主界面（如《原神》或《星穹铁道》的菜单风格）。
- **布局结构**：
  - **左侧边栏**：导航菜单（任务、地图、角色、设置）。**无商店/礼物入口。**
  - **中央主区域**：根据导航切换内容（任务列表、世界地图、剧情文本等）。
  - **右侧角色区**：大尺寸 Live2D 角色展示，可点击互动，触发对话。角色使用与桌宠模式完全一致的 AI 引擎回复。
  - **顶部状态栏**：当前世界章节标题、系统通知。**无货币/星辰币显示，无 CG 解锁提示。**

#### 3.4.2. RPG 任务系统（核心）
**用户故事 1：平凡任务的神话转化（仅用作演示，不是实际场景，不要作为底层代码写入）**
> **作为**一名大学生，**我想要**在准备期末考试时获得动力，**以便**不会拖延。  
> 我在桌宠模式下对"柚子"说："下周要考三科，好烦啊。"柚子回复："别担心，旅者。在埃瑟利亚，星之贤者即将降临三座试炼塔..."  
> 我打开软件模式，任务面板已自动生成**与当前主线章节相关联的叙事化任务**：
> - 用户任务：「星之试炼」（由"复习三科"转化而来，挂靠在当前主线第一章"觉醒的星辰"下）
>   - 子任务1：攻克「高数塔·积分迷宫」（复习第六章）—— 预估 2 个番茄钟
>   - 子任务2：突破「线代塔·向量之壁」—— 预估 1.5 个番茄钟
>   - 子任务3：解析「物理塔·电磁领域」—— 预估 2 个番茄钟
> 我在软件模式中点击子任务完成，柚子播放胜利动画，世界地图上对应塔被点亮。全部完成后，主线里程碑进度 +1，当里程碑全部达标时世界故事推进到下一章。

**任务生成、转化、追踪与结算逻辑**：
1. **输入捕获**：用户在桌宠模式的对话中表达待办事项、目标或压力来源。
2. **意图识别与转化**：通过 LLM 进行少样本提示，将用户输入解析为结构化的现实任务项，并同时生成叙事化描述和**预估时间**。
   - 请求 Prompt 包含：世界书摘要、角色设定、用户输入。
   - 要求 LLM 返回 JSON（以下为结构示例，内容字段由 AI 根据实际输入动态生成，不作为硬编码）：
     ```json
     {
       "realTask": "复习高数第六章",
       "rpgTitle": "攻破高数塔·积分迷宫",
       "rpgDescription": "在魔法学院高塔的第六层，积分迷宫守护着通往星之试炼的钥匙。你需要集中精神，彻底解析迷宫的路径。",
       "estimatedPomodoros": 2,
       "estimatedMinutes": 50,
       "subtasks": [
         { "id": "st1", "real": "阅读第六章教材", "rpg": "探索迷宫地图（阅读教材）", "completed": false },
         { "id": "st2", "real": "完成课后习题1-10", "rpg": "击败10只迷宫守卫（习题）", "completed": false }
       ]
     }
     ```
3. **用户确认与修改**：在软件模式（或桌宠模式的确认弹窗）中展示任务，用户可：
   - 修改任务标题和描述。
   - **修改 AI 预估的时间**（增加或减少番茄钟数）。
   - 增减子任务。
   - 确认后生成唯一 `taskId`，写入任务列表。
4. **追踪与提醒循环**：
   - 子任务完成可在桌宠模式的迷你面板或软件模式的主面板中勾选。
   - **时间到达时**：AI 主动询问"任务完成了吗？"，用户回答：
     - "完成了" → 标记完成，触发结算。
     - "还没" → 询问"还需要多少时间？"，用户告知后更新截止时间，进入下一轮循环。
     - "完成了一部分" → 记录当前进度，询问剩余时间，进入下一轮循环。
   - 若超过 DDL 仍未完成：触发**轻微剧情惩罚**（见 3.4.5 节），但**永远不会 Game Over**。
5. **结算与叙事推进**：当父任务所有子任务完成，触发结算：
   - 更新世界状态变量。
   - **主线进度贡献**：该用户任务完成会为主线当前章节的对应里程碑贡献进度（如 `milestone.progress += 1`）。主线章节和里程碑是预置的，用户任务只是挂靠在其下的叙事包装。
   - 检查世界书章节的里程碑完成情况，若全部完成，则推进到下一章节，解锁新背景、对话主题、角色新动作。
   - 通过 LLM 生成一段叙事反馈，描述世界变化与角色反应。
   - **无好感度数值增加**，关系深化通过叙事自然体现。
   - **标记后续追问**：任务状态变为 `completed`，并设置 `followUpPromptAt` 为完成时间 + 1~3 天（随机），届时桌宠将主动询问任务后续结果。

#### 3.4.3. 世界地图与状态可视化
- **地图展示**：以插画风格展示当前世界各区域，区域状态根据用户任务完成情况变化：
  - 未完成：灰暗、有锁链或阴影覆盖。
  - 已完成：明亮、色彩鲜艳、可能出现小建筑或雕像。
- **世界状态数值**：在地图角落显示关键世界变量（如"结晶纯度：75%"），数值随任务完成动态变化。
- **章节推进**：当前章节目标完成后，地图中央出现"前往下一章"按钮，点击进入新区域。

#### 3.4.4. 角色档案与关系系统（隐性）
- **无显式好感度条**：UI 上不显示任何数值化的好感度或进度条。
- **关系阶段**：系统内部维护一个隐性阶段（陌生→相识→熟悉→亲密→知己），由以下因素自然驱动：
  - 完成的任务总量和子任务数量。
  - 累计的番茄钟陪伴时间。
  - 对话频率和深度。
  - 连续登录天数。
- **角色档案展示**：
  - 角色背景故事、性格设定。
  - "共同记忆"时间线：记录用户与角色一起完成的重要任务和事件。
  - 已解锁的对话话题、角色新动作。
  - 没有"好感度：450/1000"这样的显示，而是显示"你们已经一起走过了 X 段旅程"。
- **解锁内容**：随着关系阶段提升，解锁：
  - 新的对话话题、更私密的故事分享。
  - Live2D 动作与表情（如"微笑"→"害羞"→"亲密眨眼"）。
  - 壁纸模式的背景与音乐变化。
  - 特殊事件触发（如角色回忆往事、写信给用户）。

#### 3.4.5. 逾期与拖延的叙事反馈（无惩罚机制）
**核心原则：永远不会 Game Over。**

任务逾期或长期拖延时，系统通过温和的叙事变化反馈给用户，而非数值惩罚（以下仅为演示文案，实际由 AI 根据上下文动态生成，不作为硬编码）：

| 完成情况 | 叙事反馈示例（演示用） |
|---------|------------|
| 顺利完成 | "列车准时抵达晨曦站。" |
| 延期完成 | "列车在暴风雪中晚点到达。" |
| 长期拖延 | "旅程进入未知区域。" |
| 多次逾期 | "星辰的轨迹发生了偏移，但前方的路依然敞开。" |

- **轻微剧情影响**：逾期可能导致世界地图上某个区域暂时笼罩薄雾、角色台词中略带担忧（但很快恢复），或任务描述发生轻微变化（如"在迷雾中探索迷宫"）。
- **无强制机制**：不会因为逾期而锁定功能、降低数值、或触发负面结局。
- **角色反应**：角色会表达关心而非责备，如"这段旅程比预想中艰难呢，需要我陪你重新规划一下吗？"

### 3.5. 模块五：状态同步与数据一致性
- **全局状态管理**：使用 Pinia 管理前端状态，主进程维护内存中的当前状态副本，通过 IPC 广播状态变更。
- **实时同步**：
  - 桌宠模式勾选子任务完成 → 立即同步到软件模式任务面板。
  - 软件模式中完成任务结算 → 关系阶段变化立即反映在桌宠模式的对话语气中。
  - 壁纸模式中的番茄钟完成 → 可在软件模式中查看专注统计（未来扩展）。
- **持久化**：所有状态变更每 5 秒自动写入 SQLite，防止崩溃丢失。

## 4. 技术规格与数据定义
### 4.1. 核心数据模型 (实体-关系)
```
User (1) ----< (N) Conversation
User (1) ----< (N) Task
User (1) ---- (1) UserStyleProfile
WorldBook (1) ----< (N) AI_Character
WorldBook (1) ---- (1) WorldState
AI_Character (1) ----< (N) Task (giver)
AI_Character (1) ---- (1) RelationshipStage (per user, 隐性)
Task (1) ----< (N) SubTask
Task (N) ---- (1) ChapterMilestone (可选)
```
详细实体字段：

- **User**
  - `id`, `username`, `createdAt`
  - `currentChapterId` (当前世界进度)
  - `activeTaskIds`: string[]
- **AICharacter**
  - `id`, `worldBookId`, `name`, `personalityTraits[]`, `background`, `speechStyle`
  - `currentMood`, `currentExpression`
  - `live2dModelPath`, `defaultCostume`
- **WorldBook**
  - `id`, `title`, `jsonDefinition` (完整 World JSON)
- **WorldState** (分离出来以便快速更新)
  - `worldBookId`, `currentChapter`, `variables` (键值对，如 `{"crystalIntegrity":75}`)
- **Task**
  - `id`, `userId`, `characterId`, `realTitle`, `rpgTitle`, `rpgDescription`
  - `estimatedPomodoros`, `estimatedMinutes` (AI 预估，用户可修改)
  - `deadline`, `status: 'active'|'completed'|'abandoned'|'overdue'`
  - `createdAt`, `completedAt`, `lastPromptAt` (上次询问完成时间)
  - `followUpPromptAt`: timestamp (任务完成后追问触发时间，随机 1-3 天)
  - `followUpCompleted`: boolean (是否已完成后续追问)
  - `followUpResult`: string (用户回答的后续结果，如"考试得了A"，供 AI 记忆)
  - `subtasks: SubTask[]`
- **SubTask**
  - `id`, `taskId`, `realDesc`, `rpgDesc`, `completed:boolean`, `progressPercent:number`
- **RelationshipStage** (隐性，不在 UI 显示)
  - `userId`, `characterId`, `stage: 'stranger'|'acquaintance'|'familiar'|'close'|'soulmate'`
  - `totalTasksCompleted`, `totalPomodoros`, `totalConversations`, `consecutiveDays`
  - `lastInteractionAt`
- **Conversation**
  - `id`, `userId`, `characterId`, `timestamp`
- **Message**
  - `id`, `conversationId`, `role` (user/assistant/system), `content`, `metadata` (动作、表情)
- **UserStyleProfile**
  - `userId`, `vector` (JSON), `lastUpdated`
- **AppState** (用于三模式状态管理)
  - `currentMode`: 'pet'|'wallpaper'|'software'
  - `petWindowBounds`: {x, y, width, height}
  - `wallpaperSettings`: {opacity, characterPosition, soundVolume}
  - `lastSyncAt`

### 4.3. 推荐技术栈与第三方服务 (Windows 桌面应用)
- **桌面框架**：Electron 28+ (主进程 Node.js, 渲染进程 Vue 3 + Vite + TypeScript)
- **状态管理**：Pinia（前端），主进程使用本地 JSON 数据库 lowdb 或 SQLite (better-sqlite3) 存储持久化数据。
- **Live2D 渲染**：Live2D Cubism SDK for Web (Cubism 4+)，在渲染进程中使用 `<canvas>` 嵌入。官方 SDK 需申请许可证，但可免费用于小型项目/比赛。考虑使用开源的 PixiJS + Live2D 插件。
- **LLM 调用**：
  - 主要方案：调用 OpenAI API (GPT-4o-mini / GPT-4o) 或 Claude API。封装统一适配器以支持切换。
  - 备选方案：本地模型通过 Ollama 或 LM Studio 提供 HTTP 接口，以降低成本并保护隐私。代码需兼容 `/v1/chat/completions` 格式。
- **语音合成 (可选，MVP 可能不包含)**：预留接口，可使用 Edge TTS 或 API。
- **任务定时提醒**：node-cron 或 Electron 内置通知。
- **白噪音播放**：HTML5 Audio API 或 Howler.js，预加载音效文件。
- **桌面宠物窗口实现**：Electron 透明无边框窗口（`transparent: true, frame: false`），置顶显示，忽略鼠标事件穿透（通过点击穿透区域控制）。
- **壁纸模式实现**：Electron 全屏无边框窗口，背景使用 CSS 半透明渐变 + 世界背景图，设置 `opacity` 和 `background-blur`。

### 4.4. 本地数据存储方案
使用 **better-sqlite3** (同步、高性能) 存储所有结构化数据：
- 表：`users`, `characters`, `worldbooks`, `tasks`, `subtasks`, `relationship_stages`, `conversations`, `messages`, `style_profile`, `world_state_variables`, `app_state`。
- 世界书 JSON 大文本可直接存储在 `worldbooks.jsonDefinition` TEXT 字段。
- 文件系统存储：Live2D 模型文件、背景图片、音效文件、日志文件。

### 4.5. 可扩展性与接口预留（V3.0 新增）
**所有 UI 美术设计、角色人设、世界书设定均可能在后续迭代中添加或修改。代码必须显式预留接口和专用文件夹，确保后续替换零成本。**

#### 4.5.1. 资源文件夹结构约定
```

│   │   ├── expressions/    # 表情配置文件（JSON 映射表情名到 Live2D 参数）
│   │   └── voice/          # 语音文件预留（如有）
│   └── {character_id}/     # 未来新角色直接新建文件夹

│   │   └── default/

│   └── fonts/              # 字体文件
├── audio/                # 音效与音乐
```

#### 4.5.2. 代码层接口预留
- **角色加载器 (`CharacterLoader`)**：
  - 通过 `characterId` 动态加载对应文件夹下的 Live2D 模型、表情配置、服装列表。
  - 接口：`loadCharacter(characterId: string): Promise<CharacterAssets>`
  - 新增角色时，只需在 `assets/characters/` 下新建文件夹并放入资源，无需修改代码。
- **主题加载器 (`ThemeLoader`)**：
  - 通过 `themeId` 动态加载 CSS 变量、颜色方案、字体、UI 图片。
  - 接口：`applyTheme(themeId: string): void`
  - 新增主题时，只需在 `assets/ui/themes/` 下新建文件夹。
- **世界书加载器 (`WorldBookLoader`)**：
  - 通过 `worldId` 动态加载 JSON 文件，注入 LLM Prompt。
  - 接口：`loadWorldBook(worldId: string): WorldBook`
  - 修改世界设定时，直接编辑 `assets/worldbooks/{world_id}.json`。
- **背景加载器 (`BackgroundLoader`)**：
  - 通过 `chapterId` 和 `sceneId` 动态加载壁纸。
  - 接口：`loadBackground(chapterId: string, sceneId?: string): string`（返回图片路径）
- **音频加载器 (`AudioLoader`)**：
  - 通过 `category` 和 `themeId` 动态加载白噪音和 BGM。
  - 接口：`playAudio(category: 'white_noise'|'bgm'|'sfx', id: string, themeId?: string)`
- **表情/动作映射表 (`ExpressionMap`, `MotionMap`)**：
  - 每个角色文件夹下包含 `expressions.json` 和 `motions.json`，定义角色可用的表情和动作名到 Live2D 参数的映射。
  - 代码中通过名称引用，不硬编码参数值。

#### 4.5.3. 配置化清单
以下所有内容必须可通过修改配置文件或替换资源文件实现，**不允许硬编码在业务逻辑中**：
- 角色名称、性格、背景故事、说话风格。
- 世界书标题、描述、规则、章节结构。
- UI 颜色方案、字体、边框样式。
- 对话气泡装饰样式。
- 番茄钟默认时长、白噪音音量。
- 主动交互频率、提醒话术模板。
- 关系阶段阈值（任务数、陪伴时间等）。
- 逾期叙事的文案库。

## 5. 非功能需求
### 5.1. 性能要求
- **AI 响应延迟**：流式输出首字延迟 < 1.5s，完整回复生成时间根据长度自适应，但需有加载动画。
- **Live2D 帧率**：在待机、壁纸、软件模式下稳定保持 60fps，CPU 占用率 < 5%（不包含 LLM 推理），GPU 占用合理。
- **内存占用**：主进程 + 渲染进程总体 < 500MB（空闲），大模型调用若使用本地模型则在另一进程，不计入。
- **主动提醒响应**：从定时器触发到气泡显示 < 200ms。
- **模式切换延迟**：桌宠 ↔ 壁纸 ↔ 软件模式切换 < 500ms，需有过渡动画。

### 5.2. 可用性与用户体验准则
- **首次引导**：新用户打开软件需经历一段简短剧情（角色被召唤），在软件模式中完成世界观介绍，并在桌宠模式下完成首次任务创建。
- **容错设计**：LLM 返回格式异常时（如不包含动作 JSON），能回退为纯文本显示并记录日志；API 掉线时桌面宠物仍可本地播放动画并提示"星辰通讯中断"。
- **无障碍**：MVP 可暂忽略，但所有气泡文字可复制。

### 5.3. 设计与美学指引
- **桌宠形象**：采用二次元 / 幻想风格，目前默认角色"柚子·月语"（银发、星纹长袍）。后续可通过关系阶段解锁新服装。
- **场景壁纸**：壁纸模式背景为对应世界章节的插画风全景（如星空下的学院），半透明模糊叠加以不干扰桌面图标。
- **软件模式 UI**：暗色星空主题，半透明面板，柔和光效，RPG 风格的边框与图标设计。
- **对话气泡**：桌宠模式采用符合角色性格的装饰（星星边缘）；壁纸模式采用侧边栏滑入式；软件模式采用底部聊天面板。
- **Live2D 动作**：动作清单需与角色性格一致，不能出现突兀的现代机械动作。三个模式下的动作集可不同（桌宠小动作、壁纸陪伴动作、软件互动动作）。
- **无数值压迫感**：UI 中不出现任何进度条、百分比、分数、等级数字。所有成长通过叙事和视觉变化体现。

## 6. 项目边界与风险
### 6.1. MVP 明确不包含的内容
- **多角色支持**：MVP 仅实现单一角色和预设的世界书，不提供多角色切换或自定义角色创建（但代码已预留接口）。
- **语音对话**：不支持语音输入输出，仅文字交互。
- **用户自定义世界书**：不开放世界编辑器，故事线由开发团队预置并可通过配置文件修改。
- **在线同步与多设备**：全部数据本地存储，无云同步。
- **社交功能**：不包含好友、排行榜或社区。
- **高级 AI Agent**：角色不会自动操控电脑或执行脚本，只是对话与提示伙伴。
- **高级用户风格学习**：MVP 仅基于简单规则与 LLM 提示注入，不训练模型或使用复杂向量库。
- **移动端**：仅支持 Windows 桌面端。
- **礼物系统与商店系统**：已删除，不在 MVP 中实现。
- **CG 解锁与相册系统**：已删除，不在 MVP 中实现。AI 生成的剧情不可预测，不预设固定 CG 解锁节点。
- **显式好感度数值**：已隐藏，UI 中不显示任何好感度条或数值。

### 6.2. 待定事项与关键讨论点
- **Live2D 模型获取**：需确定模型来源。是否使用官方免费示例模型（如 Haru）进行二次设定？或美术同学自行绘制并绑定？此为高风险依赖。代码已预留 `assets/characters/` 文件夹，替换模型即可。
- **LLM 成本与控制**：若使用外部 API，需设定令牌预算上限和请求频率限制。备选本地 LLM 的性能能否满足角色扮演及 JSON 结构化输出要求？待技术预研。
- **世界书动态演化机制**：由 LLM 直接修改世界状态变量还是本地规则引擎？前者灵活但不可控，后者稳定但僵硬。建议 MVP 先用规则+LLM 建议人工审核，待下阶段实现自动。
- **任务自动分解粒度**：用户输入一句话生成子任务可能不准确，需要交互优化。待定：是否提供模板库辅助？
- **壁纸模式与桌面交互**：壁纸模式覆盖桌面后，如何平衡"沉浸感"与"桌面操作便利性"？是否支持"点击穿透到桌面"？需测试验证。
- **资源占位与美术进度**：当前所有美术资源为占位符，需确认后续美术产出节奏，确保代码层接口与实际资源同步。

### 6.3. 已知风险与技术难点
- **桌面宠物窗口的交互与性能平衡**：透明窗口置顶、鼠标穿透区域控制、多窗口同步动画可能引发焦点管理问题，需充分测试。
- **LLM 结构化输出稳定性**：要求 LLM 在回复中嵌入动作指令和返回 JSON，存在不遵循格式的风险。必须实现健壮的解析与回退机制，并考虑使用 function calling 或 JSON mode。
- **长时间运行的内存泄漏**：桌宠常驻系统托盘，需严格管理 Live2D 模型实例和 DOM 节点，避免资源泄漏。
- **上下文窗口成本**：随着世界书增大、对话历史增长，输入 token 消耗巨大。需设计高效的摘要记忆模块与 RAG（检索增强生成）预案。
- **三模式状态同步复杂性**：确保三种模式下的 Live2D 实例、音频播放、定时器等资源不冲突，模式切换时正确释放和重建资源。
- **Vibe Coding 的代码质量**：AI 生成代码需经过代码审查与重构，避免不可维护的"意大利面条式"代码。本 PRD 将作为测试用例生成的依据。
- **资源动态加载的异步处理**：角色、主题、世界书等资源的动态加载需处理异步边界和错误回退（如资源缺失时显示默认占位图）。
